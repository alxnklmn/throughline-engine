# Throughline

**Движок человеческой непрерывности для AI-ассистентов.**

Сделано для того, чтобы ИИ мог присутствовать в жизни человека, не превращаясь в слежку.

## Сдвиг

Большинство AI-ассистентов **реактивны**. Ты спросил — они ответили — диалог умер. Но человек не устроен как набор изолированных запросов. Человек говорит: *«завтра экзамен, я в панике»* — и уходит. Не потому что тема закрыта. А потому что жизнь продолжилась.

Реактивный ассистент ничего не делает. Шедулерный ассистент шлёт напоминание. Ассистент с **Throughline** задаёт правильный вопрос:

> Осталась ли здесь человеческая нить, к которой стоит бережно вернуться — уместно, с разрешения?

Если да — он возвращается. Если нет — **молчит.**

Это молчание не баг. Это feature, у которого есть собственная метрика (Silence Correctness — правильность молчания). Throughline — это слой, который позволяет ассистенту *присутствовать* в жизни человека, не превращаясь в слежку, CRM или очередной завод уведомлений.

---

## Что он делает

- **Извлекает** Human State Threads из диалога — тревоги, обещания, важные события, намерения, творческие импульсы
- **Оценивает** возможности для инициативы через строгую модель **veto-first** — согласие бинарно, его нельзя «перевесить» важностью
- **Составляет** заботливые сообщения, которые ощущаются как друг с разумной памятью, а не как база данных с идеальным recall (через намеренную генерализацию)
- **Калибруется** через обратную связь — учится, когда инициатива была желанна, а когда нет, по каждому пользователю и категории
- **Молчит правильно** — и измеряет это как основную метрику успеха, а не как побочный эффект

## Что он отказывается делать

- Оптимизироваться на engagement, retention, длительность сессии или любые другие метрики экономики внимания
- Переносить контекст между отношениями — то, что Маша рассказала боту, остаётся в канале Маши и никогда не утекает к кому-то ещё (см. [Неприкосновенные инварианты](#три-неприкосновенных-инварианта))
- Заявлять о человеческих чувствах — никаких *«я переживал за тебя»*, никаких *«я скучал»*. Он демонстрирует внимательность поведением, никогда не утверждает эмоции
- Хранить что-либо в открытом виде — care-данные это самые чувствительные данные в системе
- Отправлять сообщения в high-sensitivity категориях без явного согласия
- Звонить домой — нулевая аналитика, нулевой телеметрии, никаких «улучшений на основе использования»

## Быстрый старт

```bash
pip install throughline
```

```python
from throughline import Engine, FeedbackType
from throughline.types import CategoryConsent
from datetime import datetime

# Один engine на одного владельца. Per-relationship isolation
# гарантируется на уровне storage API.
engine = Engine(
    storage_path="~/.throughline/alex.db",
    encryption_key_source="keychain",   # ключ SQLCipher живёт в OS keychain
    owner_id="alex",
)

# Владелец явно разрешает, в каких категориях возможны follow-up'ы.
# Без согласия каждый veto срабатывает — движок молчит.
engine.set_consent(
    owner_id="alex",
    category="study",
    level=CategoryConsent.EVENT,
    quiet_hours=(23, 9),
)

# Каждое сообщение в пайплайне твоего бота скармливаем движку.
engine.observe_message(
    owner_id="alex",
    contact_id="masha",
    direction="self",                   # владелец → ассистент
    text="завтра экзамен, паника",
    timestamp=datetime.utcnow(),
)

# Периодически опрашиваем движок на решения (раз в 5–15 минут типично).
for decision in engine.tick(owner_id="alex"):
    if decision.level == "direct":
        await your_bot.send(decision.composed_message)
    # Другие уровни (passive_prompt, soft_inject, status_nudge) — в v0.2.

# Всегда замыкай feedback loop. Без него движок не может калиброваться.
engine.record_feedback(
    decision_id=decision.id,
    feedback_type=FeedbackType.ENGAGED,
    raw_signal=user_reply_text,
)
```

## Три неприкосновенных инварианта

Это не аспирационные принципы. Это архитектурные правила, зашитые на уровне storage, а не policy — так что баги в верхних слоях не могут их сломать. Имплементации, нарушающие любой из них, не являются валидными имплементациями Throughline.

### I-1. Изоляция по отношениям

Информация, которой человек X поделился с владельцем Y, попадает и остаётся в канале `(Y, X)`. Она никогда не доходит до кого-то ещё. Никогда. Ни по какой причине.

> *Маша рассказала боту, что развелась. Через три дня бот помогает Алексу составить сообщение для Васи. Бот **не должен** упоминать развод Маши — даже косвенно, даже помогая, даже если Алекс спросил «что нового у моих контактов». Информация из канала Алекс-Маша не существует в канале Алекс-Вася.*

Это и есть граница. В публичном API просто нет метода, который позволил бы её пересечь.

### I-2. Суверенитет владельца

Человек, которому служит ассистент, имеет абсолютную, мгновенную, необратимую власть над всеми хранимыми о нём continuity-данными:

- Полный экспорт (JSON) в любой момент
- Полное стирание (один клик, без восстановления) в любой момент
- Удаление per-thread, per-category
- Право навсегда отключить любую категорию

### I-3. Шифрование at rest

Continuity-данные содержат эмоциональные паттерны пользователя, его уязвимости, тревоги, личные вопросы. Все таблицы, которыми управляет Throughline, ОБЯЗАТЕЛЬНО зашифрованы at rest (SQLCipher или эквивалент), ключ хранится в защищённом хранилище операционной системы. Plaintext-хранение — не валидная конфигурация.

## Архитектура в одном экране

```
                ┌────────────────────────────────────┐
                │      Host Application (твой бот)   │
                └──────────────┬─────────────────────┘
                               │
                               ▼ (события)
        ┌──────────────────────────────────────────┐
        │            Throughline Engine            │
        │                                          │
        │  Extraction        — находит нити        │
        │  Continuity Memory — per-pair изолирован │
        │  Initiative Eval   — vetoes → multi → +  │
        │  Generalization    — гасит точность      │
        │  Composer          — составляет сообщ.   │
        │  Feedback Loop     — учится калибровке   │
        └────────────────┬─────────────────────────┘
                         │
                         ▼ (решения)
                ┌────────────────────────────────────┐
                │      Host Application              │
                └────────────────────────────────────┘
```

Host application отвечает за:
- Пайплайн сообщений (фидит события на вход)
- Поверхность доставки (действует по решениям на выходе)
- Контролы для владельца (UI согласий, кнопки export/wipe)

Throughline отвечает за:
- Решение, инициировать ли, когда и с каким уровнем интрузивности
- Как выразить заботу, не переходя в слежку
- Гарантию изоляции по отношениям
- Калибровку, которая удерживает систему от деградации в шум

Полная архитектурная детализация — [SPEC.md](./SPEC.md).

## Care Score

Решения вычисляются в фиксированном порядке. Порядок стадий не подлежит обсуждению — он кодирует разницу между заботой и преследованием.

```
1. VETOES         — бинарные блокеры (consent, sensitivity, rebuff, crisis, ...)
                   → любой veto срабатывает → SILENCE, дальше не считаем
2. MULTIPLIERS    — ситуативное масштабирование (когнитивная нагрузка, темп, ...)
                   → произведение ниже порога → SILENCE
3. LINEAR SCORE   — importance + emotion + timing + usefulness
4. FINAL          — base_score × произведение_multipliers
5. GRADUATION     — final мапится в один из пяти уровней интрузивности:
                   SILENCE / PASSIVE / SOFT_INJECT / NUDGE / DIRECT
```

Самая распространённая ошибка в care-системах — делать consent весом, а не veto. Высокое значение importance никогда не должно перебивать «пользователь сказал не спрашивать про это». Эта асимметрия — то, что отделяет друга от CRM. См. [SPEC.md §5](./SPEC.md#5-the-care-score) для обоснования.

## Уровни инициативы

Движок не считает инициативу бинарной. Он выбирает *самый низкий* уровень интрузивности, который всё ещё служит нити:

| Уровень | Что делает | Срабатывает при |
|---|---|---|
| **Silence** | ничего | дефолт; предпочтительнее, чем неуверенное действие |
| **Passive prompt** | видно только если владелец откроет dashboard/Mini App | score ≥ 0.40 |
| **Soft inject** | вплетается в следующее исходящее сообщение естественно | score ≥ 0.55 |
| **Status nudge** | меняет видимый статус, без сообщения | score ≥ 0.70 |
| **Direct message** | полное сообщение с уведомлением | score ≥ 0.85 |

v0.1 реализует только direct messages. Нижние уровни приходят с v0.2.

## Генерализация, или: почему точный recall ощущается жутко

Друг помнит обобщённо. *«Твой экзамен по физике в 14:30 только что закончился — как?»* — это фраза сталкера. *«У тебя сегодня было что-то важное — как прошло?»* — это фраза друга.

Throughline прогоняет каждый факт через слой `generalize()` перед композицией:

| Сырой факт | Генерализованный |
|---|---|
| `exam at 14:30 physics Hall B` | *у тебя сегодня было что-то важное* |
| `lost appetite, last meal Wednesday 11pm` | *ты говорил, что с едой было непросто* |
| `said you'd watch The Brutalist at 8pm` | *тот фильм, который ты собирался посмотреть* |

Три уровня: `literal` (только для UI-отображения), `loose` (по умолчанию), `oblique` (для high-sensitivity нитей).

## Статус проекта

**v0.1 — спецификация написана, имплементация MVP в процессе.**

Что работает сегодня:
- Полный veto layer (10 vetoes, полное тестовое покрытие) — `throughline/vetoes.py`
- Полный scoring layer (multipliers, linear addends, graduated initiation) — `throughline/scoring.py`
- Генерализация со статическими паттернами — `throughline/generalization.py`
- Public type system (`Thread`, `Decision`, `FeedbackType` и т.д.) — `throughline/types.py`
- Полная архитектурная спека — `SPEC.md`

Что в стабах (следующая имплементационная работа):
- `Engine` façade — SQLCipher storage, message extraction, evaluation loop, feedback persistence
- LLM-based генерализация (v0.2)
- Нижние уровни инициативы (v0.2)
- Soul Graph / долгосрочная личность (v0.4)

## Контрибьютинг

PR'ы welcome под необсуждаемыми ограничениями:

- **Никаких послаблений неприкосновенных инвариантов.** [SPEC.md §3](./SPEC.md#3-inviolable-invariants) — это граница. PR'ы, ослабляющие любой инвариант, будут закрыты.
- **Никаких engagement-оптимизаций.** Любой PR, чья заявленная цель — «увеличить объём сообщений / длительность сессии / sticky» — это не тот проект.
- **Privacy-affecting изменения требуют явного review** с письменным обоснованием.
- **Style**: `black` + `ruff`, conventional commits.
- **Тесты** обязательны для: vetoes, изоляции, feedback loop.

Throughline касается самых личных уголков жизни пользователей. От контрибьюторов ожидается воспринимать это всерьёз. Полные детали — [SPEC.md §17](./SPEC.md#17-contributing).
